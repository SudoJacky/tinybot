# Desktop WebUI Parity Inventory

This document is the phase-1 routing artifact for `mirror-webui-in-desktop-app`
and the foundation inventory for `desktop-native-ux-adaptation`. The desktop app
hosts the root WebUI as the feature source of truth while desktop-native
workbench containers are introduced behind explicit gates. Keep this inventory
current when a root WebUI module adds a route, modal, static asset, gateway
contract, shared helper, or desktop adaptation boundary that the desktop shell
must preserve.

## Root WebUI Source Inventory

| Surface | Source of truth | Desktop parity contract |
| --- | --- | --- |
| HTML shell and page metadata | `webui/index.html` | Install the root body, `lang`, `data-theme`, local stylesheet links, and local icon links before importing `/assets/src/main.js`. |
| App startup and state | `webui/assets/src/main.js`, `webui/assets/src/state.js`, `webui/assets/src/legacy/app.js` | Import the root WebUI entrypoint after desktop gateway and render globals are installed. |
| Styles and image assets | `webui/assets/styles.css`, `webui/assets/styles/**`, `webui/assets/logo*.svg`, `webui/assets/social-preview.svg` | Serve and bundle `/assets/**` from root `webui/assets`; leave those requests on the desktop static origin. |
| Docs pages | `webui/docs/index.html`, `cli.html`, `config.html`, `gateway.html`, `knowledge.html`, `providers.html`, `quickstart.html`, `skills.html`, `tasks.html`, `tools.html`, `webui.html` | Serve and bundle `/docs` plus `/docs/**` from root `webui/docs`; leave docs requests on the desktop static origin. |
| i18n dictionaries | `webui/assets/src/i18n/en-US.js`, `webui/assets/src/i18n/zh-CN.js`, `webui/assets/src/i18n/index.js` | Load the same dictionaries through `/assets/src/i18n/**` and verify language switching updates static and dynamic labels. |
| Shared WebUI helpers | `agent-ui-events.js`, `cowork-chat.js`, `knowledge-traceability.js`, `provider-cards.js`, `utils/**` | Serve helper modules through `/assets/src/**`; exclude root WebUI `*.test.*` files from desktop build output. |

## Desktop-native Adaptation Boundary

| Layer | Ownership | Desktop adaptation rule |
| --- | --- | --- |
| Gateway contracts | Root WebUI and backend | Preserve endpoint paths, request payloads, auth refresh, and response semantics for chat, config, providers, knowledge, tools, skills, workspace, and Cowork. |
| Shared feature helpers/controllers | Root `webui/assets/src/**` | Extract or wrap reusable state projection, render helpers, i18n, validation, and action request builders before mounting them in desktop containers. |
| Browser page layout | Root `webui/index.html`, `webui/assets/src/legacy/app.js`, root CSS | Keep as the recoverable parity route. Do not require browser layout modals or full-page shell structure for native desktop containers. |
| Desktop containers | `apps/desktop/src/**` | Own Tauri chrome, workbench regions, panel persistence, command routing, OS file/opener/menu/notification boundaries, gateway lifecycle UI, and runtime status surfaces. |
| Verification evidence | `apps/desktop` tests, root WebUI tests, Tauri smoke notes | Mark desktop-native modules complete only when automated adapter coverage and real desktop/manual evidence are both named. |

## Foundation Audit for Desktop-native UX

| Area | Current finding | Migration implication |
| --- | --- | --- |
| Desktop entrypoint | `apps/desktop/src/desktopBootstrap.ts` starts or attaches to the gateway, installs bridge/render globals, injects root `webui/index.html`, reinstalls the Tauri frame, then imports `/assets/src/main.js`. | Native workbench slices should branch before root body injection, but must keep the root injection path callable for recovery and parity checks. |
| Root WebUI injection | `installWebUiShell` copies root `lang`, `data-theme`, local stylesheet/icon links, strips scripts, and replaces the desktop startup body with root WebUI body nodes. | Desktop containers may reuse root assets and helpers without copying root HTML wholesale; the root shell remains the fallback route. |
| Fallback desktop modules | `main.ts`, `desktopChatSessionController.ts`, `nativeChat.ts`, `agentUiEvents.ts`, gateway HTTP/WebSocket clients, and status view modules remain tested but are not the default user route. | Treat them as adapter/reference material for future desktop panes, not as an independent product surface with separate API semantics. |
| `legacy/app.js` coupling | The module still binds chat, sessions, modals, knowledge, tools, skills, workspace, config/providers, Cowork, help tour, and keyboard handlers directly to global `state`, `elements`, `document`, `window`, modal DOM, and local storage. | Extract shared controllers/helpers module-by-module before desktop containers mount those workflows; avoid moving the entire legacy page into desktop TypeScript. |
| Recoverability | Native workbench is the desktop startup default; root WebUI can still be forced with `?desktop-workbench=root` even after a stored native preference exists. | During phased rollout, a broken or incomplete native workbench path can be bypassed without changing gateway or root WebUI behavior. |
